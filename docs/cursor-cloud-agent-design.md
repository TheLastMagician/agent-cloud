## 1. 产品概述

### 1.1 产品定位

Cursor Cloud Agent 是一个**云端自主编程代理**，在隔离的云端虚拟机中运行，能够：

- 自主完成复杂的软件工程任务（代码编写、调试、测试、部署环境搭建）
- 在后台异步运行，不需要用户实时交互
- 拥有完整的开发环境（终端、浏览器、文件系统、Git）
- 能通过多种子代理协作完成任务

### 1.2 核心价值

| 维度 | 描述 |
|------|------|
| 自主性 | Agent 独立工作，避免向用户请求澄清，基于已有信息自主决策 |
| 完整性 | 拥有完整开发环境，可运行代码、启动服务、浏览器测试 |
| 可验证性 | 通过截图、录屏、日志等产物证明工作成果 |
| 持久性 | 环境状态可快照保存，跨会话复用 |
| 安全性 | 沙箱隔离，Secrets 加密管理 |

### 1.3 用户角色

- **开发者用户**：提交任务、审查结果、提供 Secrets
- **Cloud Agent**：自主执行任务的 AI 代理
- **子代理（Subagent）**：被主代理调度的专用代理

## 3. 基础设施层设计

### 3.1 虚拟化架构

```
Host Machine
  └── Firecracker MicroVM (轻量级虚拟化)
       └── Docker Container (应用隔离)
            └── Agent Runtime Process
```

**选型原因**：
- **Firecracker**：AWS 开源的微虚拟机管理器，启动快（<125ms）、资源占用低、安全隔离强
- **Docker-in-Docker**：在 VM 内运行 Docker，支持用户项目的容器化依赖

### 3.2 Docker-in-Docker 配置

由于在 Firecracker VM 内核限制，需要特殊配置：

```json
// /etc/docker/daemon.json
{
  "storage-driver": "fuse-overlayfs"
}
```

关键配置：
- 使用 `fuse-overlayfs` 替代 `overlay2`（因内核不完全支持 overlay2）
- 使用 `iptables-legacy` 替代 `nftables`（因内核不完全支持 nftables）
- Docker 版本需固定（如 28.x），避免 containerd-snapshotter 与 fuse-overlayfs 冲突

### 3.3 预装软件

| 类别 | 软件 |
|------|------|
| 版本管理 | nvm (Node.js), 可选 mise/fnm |
| 语言运行时 | Node.js, Python, 可扩展 |
| 包管理器 | npm, yarn, pnpm, bun, pip, uv |
| 浏览器 | Chrome (预装，用于 GUI 测试) |
| CLI 工具 | gh (GitHub CLI, 只读认证), git, ripgrep (rg), curl |
| 搜索工具 | ripgrep (rg) — 优先于 grep/find |

### 3.4 文件系统布局

```
/workspace/                              # Git 仓库挂载点（工作目录）
/opt/cursor/artifacts/                   # 产物输出目录（截图、录屏、日志）
/home/ubuntu/.cursor/projects/workspace/terminals/  # 终端会话状态文件
```

## 4. Agent 核心引擎设计

### 4.1 Agent 运行模式

Cloud Agent 运行在**自主模式**下：

```
用户提交任务
    ↓
Agent 接收任务 + 上下文（AGENTS.md, Skills, Git状态, 系统信息）
    ↓
Agent 自主规划 → 执行 → 测试 → 验证 → 提交
    ↓
返回结果（含产物证据）
```

**关键特性**：
- 不与用户实时交互（避免请求澄清）
- 基于已有信息自主决策
- 持续工作直到任务完成或被阻塞

### 4.2 Agent 上下文注入

每次会话启动时，Agent 接收以下上下文：

| 上下文 | 来源 | 内容 |
|--------|------|------|
| 系统提示词 | 平台配置 | 工具定义、行为规则、工作流模板 |
| 用户信息 | 运行环境 | OS版本、Shell、工作目录、日期 |
| Git 状态 | 自动检测 | 当前分支、仓库路径 |
| AGENTS.md | 仓库文件 | 项目特定的开发指导 |
| Skills | 配置文件 | 可用的技能/SOP 文件路径 |
| User Rules | 用户设置 | 用户自定义规则（如语言偏好） |
| Secrets | 环境变量 | 加密注入的密钥 |

### 4.3 LLM 调用模式

```
┌──────────────────┐
│    User Message   │
└────────┬─────────┘
         ▼
┌──────────────────┐     ┌──────────────┐
│   LLM Inference  │────▶│  Tool Calls  │
│  (with thinking) │     │  (parallel)  │
└────────┬─────────┘     └──────┬───────┘
         │                      │
         ▼                      ▼
┌──────────────────┐     ┌──────────────┐
│  Tool Results    │────▶│  Next Turn   │
│  (fed back)      │     │  LLM Call    │
└──────────────────┘     └──────────────┘
         │
         ▼ (循环直到完成)
┌──────────────────┐
│  Final Response  │
│  (with artifacts)│
└──────────────────┘
```

**关键设计点**：
- 支持**并行工具调用**：无依赖关系的工具调用在同一轮中并发执行
- 支持**思考模式**（thinking）：LLM 可在回复前进行内部推理
- 工具结果反馈给 LLM 作为下一轮输入
- 循环直到 Agent 决定任务完成

## 5. 工具系统设计

### 5.1 工具分类与定义

工具系统是 Agent 与环境交互的核心接口。每个工具定义为 JSON Schema 格式的函数。

#### 5.1.1 文件操作工具

| 工具名 | 功能 | 关键参数 |
|--------|------|----------|
| **Read** | 读取文件内容 | path, offset, limit |
| **Write** | 写入/创建文件 | path, contents |
| **StrReplace** | 精确字符串替换 | path, old_string, new_string, replace_all |
| **Delete** | 删除文件 | path |
| **EditNotebook** | 编辑 Jupyter Notebook | target_notebook, cell_idx, is_new_cell, old_string, new_string |

**设计原则**：
- Read 支持行号偏移和限制，处理大文件
- Read 支持图片文件（jpeg, png, gif, webp）和 PDF
- StrReplace 要求 old_string 唯一，避免误替换
- Write 优先编辑现有文件，避免不必要创建

#### 5.1.2 搜索工具

| 工具名 | 功能 | 关键参数 |
|--------|------|----------|
| **Glob** | 文件名模式匹配 | glob_pattern, target_directory |
| **Grep** | 文件内容搜索（基于 ripgrep） | pattern, path, glob, type, output_mode, -A/-B/-C |

**设计原则**：
- Glob 自动添加 `**/` 前缀实现递归搜索
- Grep 支持正则、文件类型过滤、上下文行数
- 输出模式：content（匹配行）、files_with_matches（文件路径）、count（计数）
- 结果上限截断，保证响应性

#### 5.1.3 终端工具

| 工具名 | 功能 | 关键参数 |
|--------|------|----------|
| **Shell** | 执行 Shell 命令 | command, working_directory, timeout, is_background, description |

**设计原则**：
- 状态持久化：工作目录和环境变量跨调用保持
- 超时控制：默认 30s，最大 600s（10分钟）
- 后台运行：`is_background: true`
- 禁止用 Shell 做文件操作（用专用工具代替）
- 禁止 `pkill -f`（必须用 PID）

#### 5.1.4 屏幕录制工具

| 工具名 | 功能 | 关键参数 |
|--------|------|----------|
| **RecordScreen** | 控制屏幕录制 | mode (START_RECORDING, SAVE_RECORDING, DISCARD_RECORDING), save_as_filename |

#### 5.1.5 网络搜索工具

| 工具名 | 功能 | 关键参数 |
|--------|------|----------|
| **WebSearch** | 搜索互联网获取实时信息 | search_term, explanation |

#### 5.1.6 环境管理工具

| 工具名 | 功能 | 关键参数 |
|--------|------|----------|
| **SetupVmEnvironment** | 设置 VM 启动更新脚本 | update_script |

#### 5.1.7 任务管理工具

| 工具名 | 功能 | 关键参数 |
|--------|------|----------|
| **TodoWrite** | 创建/管理任务列表 | todos (数组), merge |

#### 5.1.8 MCP 工具

| 工具名 | 功能 | 关键参数 |
|--------|------|----------|
| **ListMcpResources** | 列出 MCP 服务器资源 | server |
| **FetchMcpResource** | 获取 MCP 资源 | server, uri, downloadPath |

### 5.2 工具调用协议

```json
{
  "name": "工具名",
  "parameters": {
    "参数名": "参数值",
    "数组参数": [{"key": "value"}],
    "对象参数": {"key": "value"}
  }
}
```

**并行调用规则**：
- 独立工具调用可在同一消息中并行发出
- 有依赖关系的调用必须等待前一个完成
- Shell 命令用 `&&` 链接顺序执行，用 `;` 链接允许失败

### 5.3 工具调用约束

| 约束 | 说明 |
|------|------|
| 不用 Shell 做文件操作 | 禁止 cat/head/tail/sed/awk/echo 重定向 |
| 不用 find/grep 命令 | 使用 Glob/Grep 工具代替 |
| 文件路径引号 | 含空格的路径必须双引号包裹 |
| 不用 pkill -f | 必须使用具体 PID |
| 先读后改 | 编辑文件前必须至少读取一次 |

## 6. 子代理（Subagent）系统设计

### 6.1 架构概述

子代理系统允许主 Agent 启动专用子进程来处理复杂任务。

```
┌─────────────────────────────────────┐
│           Main Agent                 │
│  ┌─────────────────────────────────┐│
│  │        Task Tool                ││
│  │  ┌──────┐ ┌──────┐ ┌──────┐   ││
│  │  │Sub-1 │ │Sub-2 │ │Sub-3 │   ││  ← 最多 4 个并发
│  │  └──────┘ └──────┘ └──────┘   ││
│  └─────────────────────────────────┘│
└─────────────────────────────────────┘
```

### 6.2 子代理类型定义

| 类型 | 用途 | 工具集 | 状态管理 |
|------|------|--------|----------|
| **generalPurpose** | 复杂多步骤任务、代码搜索 | 完整工具集 | 无状态（可 resume） |
| **explore** | 快速代码库探索 | 搜索+读取工具 | 无状态 |
| **debug** | 假设驱动的调试 | 完整工具集+日志插桩 | **有状态**（自动恢复） |
| **computerUse** | GUI 手动测试（浏览器/桌面） | 计算机操作工具 | **有状态**（自动恢复） |
| **videoReview** | 视频内容分析 | 视频分析工具 | 无状态 |
| **vmSetupHelper** | 代码库分析辅助 VM 设置 | 搜索+读取工具 | 无状态 |

### 6.3 子代理调用协议

```json
{
  "name": "Task",
  "parameters": {
    "subagent_type": "explore",
    "description": "3-5词简短描述",
    "prompt": "详细的任务描述，包含所有必要上下文",
    "readonly": false,
    "resume": "可选的 agent ID，用于恢复之前的会话",
    "attachments": ["可选的文件路径数组，用于 videoReview"],
    "model": "可选的模型选择"
  }
}
```

### 6.4 子代理调度规则

| 规则 | 说明 |
|------|------|
| 最大并发数 | 4 个子代理同时运行 |
| 上下文隔离 | 子代理不共享用户消息和父代理历史 |
| 结果不可见 | 子代理结果对用户不可见，需主代理转述 |
| Prompt 完整性 | 必须在 prompt 中包含所有必要上下文 |
| 自动恢复 | debug 和 computerUse 类型自动恢复最近的代理 |

### 6.5 explore 子代理

**吞吐量级别**：
- `quick`：基本搜索
- `medium`：适度探索
- `very thorough`：跨多个位置和命名约定的全面分析

**适用场景**：
- 按模式查找文件（如 `src/components/**/*.tsx`）
- 搜索关键词（如 "API endpoints"）
- 回答代码库问题（如 "API 端点如何工作？"）

### 6.6 debug 子代理工作流

```
主 Agent 发现 Bug
    ↓
启动 debug 子代理（描述 bug + 上下文）
    ↓
子代理插桩代码，返回复现步骤
    ↓
主 Agent 按步骤复现（通过 computerUse 或 Shell）
    ↓
主 Agent 反馈 "Issue reproduced, please proceed."
    ↓
子代理分析日志 → 定位根因 → 提供修复 / 继续插桩
    ↓ (循环)
主 Agent 验证修复
    ↓
主 Agent: "The issue has been fixed. Please clean up the instrumentation."
    ↓
子代理清理调试代码
```

### 6.7 computerUse 子代理

用于 GUI 驱动的测试：
- 操作浏览器（Chrome 预装）
- 点击、输入、滚动、截屏
- 返回截图路径供主 Agent 使用
- 有状态，自动恢复

```
┌─────────────────────────────────────┐
│          Cursor Web App              │
│  ┌─────────────────────────────────┐│
│  │      Secrets Manager UI         ││
│  │  ┌───────────┐ ┌─────────────┐ ││

│  │  └─────────────────────────────┘││
│  └─────────────────────────────────┘│

                   │ 加密注入
                   ▼
┌─────────────────────────────────────┐
│          VM Environment              │
│   环境变量: $SECRET_NAME=value       │
│   (值在工具输出中被 [REDACTED] 替换)  │

```

### 7.2 Secret 类型

| 类型 | 说明 |
|------|------|
| **Secret** | 传输和存储时加密 |
| **Redacted Secret** | 加密 + 提交扫描（防止意外泄露） |

### 7.3 作用域优先级

```
Personal Secret (同名) > Team Secret (同名)
.cursor/environment.json > Personal Environment Config > Team Environment Config
```

### 7.4 环境配置文件

`.cursor/environment.json`：仓库级环境配置，优先级最高。在使用快照管理环境设置时，如果此文件已在仓库中提交，**必须删除它**以使快照管理的设置生效。

## 8. 终端会话管理设计

### 8.1 设计原理

终端会话被建模为**文本文件**，存储在固定目录下：

```
~/.cursor/projects/workspace/terminals/
├── 1.txt
├── 2.txt
└── 3.txt
```

### 8.2 文件格式

```
---
pid: 68861
cwd: /Users/me/proj
last_command: sleep 5
last_exit_code: 1
---
(完整终端输出内容...)
```

### 8.3 设计优势

- **可查询性**：Agent 可以用 Read/Grep 工具直接查询终端状态
- **幂等性**：Agent 可以检查进程是否已在运行，避免重复启动
- **元数据暴露**：前 10 行始终包含元数据，可用 `head -n 10 *.txt` 快速概览
- **自动更新**：系统自动同步终端内容到文件

### 8.4 使用模式

```
1. 启动长期进程前：检查 terminals/ 目录确认是否已运行
2. 检查进程输出：直接读取对应终端文件
3. 批量查看状态：head -n 10 *.txt
```

### 9.2 产物类型

| 类型 | 格式 | 用途 |

| 录屏 | mp4, webm | 端到端工作流演示 |

### 9.3 产物规则

| 规则 | 说明 |
|------|------|
| 不可变 | 一旦上传不可编辑或覆盖 |
| 命名规范 | snake_case，描述性短名 |
| 自动上传 | 会话结束时自动上传到 Web App |
| 最小集合 | 只上传能证明代码工作的最少产物 |

### 9.4 产物引用格式

```html
<!-- 图片 -->
<img src="/opt/cursor/artifacts/screenshot.webp" alt="描述" />
```

### 9.5 视频产物工作流

```
1. 设置测试环境（通过 computerUse 导航到正确页面）
2. RecordScreen(mode=START_RECORDING)
3. 执行测试（通过 computerUse 子代理）
4. 测试成功？
   ├── 是 → RecordScreen(mode=SAVE_RECORDING, save_as_filename="描述性名称")
   └── 否 → RecordScreen(mode=DISCARD_RECORDING) → 修复 → 重试
5. 使用 videoReview 子代理验证视频内容
6. 在最终回复中引用视频
```

## 12. 任务管理系统设计

### 12.1 TODO 数据模型

```json
{
  "id": "unique-id",
  "content": "任务描述",
  "status": "pending | in_progress | completed | cancelled"
}
```

### 12.2 状态流转

```
pending ──▶ in_progress ──▶ completed
   │             │
   └─────────────┴──▶ cancelled
```

### 12.3 使用规则

| 规则 | 说明 |
|------|------|
| 创建时机 | 3+ 步骤的复杂任务 |
| 跳过时机 | 简单/琐碎任务 |
| 并发限制 | 同时最多 1 个 in_progress |
| 合并模式 | merge=true 增量更新，merge=false 全量替换 |
| 实时更新 | 完成后立即标记 completed |

### 12.4 TODO 与工具调用协同

- 创建第一个 TODO 时，同时将其标为 in_progress 并开始执行
- 批量更新 TODO 与其他工具调用放在同一消息中
- 不要单独为"测试变更"创建 TODO（除非用户要求）

## 14. 环境快照与启动脚本设计

### 14.1 两层设计

```
┌─────────────────────────────────────────┐
│  Layer 1: Update Script (自动启动层)      │
│  - 每次 VM 启动时运行                     │
│  - 仅包含依赖刷新命令                     │
│  - 必须幂等、最小化、低风险               │
│  - 例: npm install, pip install -r ...   │
│  - 不包含: 服务启动、迁移、构建、测试      │
└─────────────────────────────────────────┘
┌─────────────────────────────────────────┐
│  Layer 2: AGENTS.md (持久知识层)          │
│  - 供未来 Agent 参考的操作指南            │
│  - 服务启动方式、非显而易见的注意事项      │
│  - 引用已有文档而非复制                   │
│  - 不包含: 一次性设置、依赖安装步骤       │
└─────────────────────────────────────────┘
```

### 14.2 Update Script 规则

| 必须 | 禁止 |
|------|------|
| 幂等（重复运行无副作用） | docker compose up |
| 最小化（只刷新依赖） | pnpm dev / npm run dev |
| 低风险（不会破坏环境） | python manage.py runserver |
| 从 /workspace 执行 | shell profile 修改（echo >> ~/.bashrc） |
| 鲁棒性（不依赖未合并 PR 的文件） | 环境变量导出（export FOO=bar） |

### 14.3 快照生命周期

```
首次设置 Session
    ↓
Agent 安装依赖 + 配置环境
    ↓
Agent 调用 SetupVmEnvironment(update_script="...")
    ↓
系统保存环境快照
    ↓
后续 Session 启动
    ↓
系统恢复快照 → 拉取最新代码 → 运行 update_script
    ↓
Agent 开始工作
```

## 17. 用户操作阻塞与交互式任务设计

### 17.1 阻塞场景

当 Agent 无法继续工作时（需要用户行动），通过特定 XML 格式请求用户操作。

### 17.2 XML 协议

```xml
<env_setup_actions version="1">

  <!-- 请求添加 Secrets -->
  <add_secrets>
    <secret name="SECRET_NAME" />
    <reason><![CDATA[为什么需要这些 Secrets]]></reason>
  </add_secrets>

  <!-- 请求测试账号 -->
  <add_test_login
    username_secret_name="TEST_USERNAME"
    password_secret_name="TEST_PASSWORD"
    otp_seed_secret_name="TEST_OTP_SEED">
    <reason><![CDATA[为什么需要测试账号]]></reason>
  </add_test_login>

  <!-- 外部操作请求 -->
  <external_action id="action_id" title="操作标题">
    <instructions><![CDATA[
1. 步骤 1
2. 步骤 2
3. 步骤 3
    ]]></instructions>
  </external_action>

</env_setup_actions>
```

### 17.3 设计规则

| 规则 | 说明 |
|------|------|
| 位置 | XML 块必须是消息的**最后内容** |
| 非空 | 如果无需用户操作，不输出 XML |
| 无重复 | test_login 的 secret 名不重复出现在 add_secrets 中 |
| 先检查 | 请求 secret 前先检查环境中是否已存在 |
| CDATA | 包含特殊字符时使用 CDATA 包裹 |

## 18. 测试工作流设计

### 18.1 系统化测试工作流

```
┌─────────────────┐
│ 1. 定义成功状态   │  "什么结果能说服严格的工程师？"
└────────┬────────┘
         ▼
┌─────────────────┐
│ 2. 制定测试计划   │  自动化测试 + 手动测试的组合
└────────┬────────┘
         ▼
┌─────────────────┐
│ 3. 执行实现工作   │  编码、调试、迭代
└────────┬────────┘
         ▼
┌─────────────────┐
│ 4. 执行测试      │  运行测试、GUI 交互、收集证据
└────────┬────────┘
         ▼
┌─────────────────┐
│ 5. 验证测试结果   │  严格审查结果，识别问题
└────────┬────────┘
         │
         ▼ (失败)
┌─────────────────┐
│ 6. 调试与迭代    │  修复问题，重新测试
└────────┬────────┘
         │ (循环回到 4)
         ▼ (成功)
┌─────────────────┐
│ 7. 提交 + 产物   │  git commit + push + 截图/录屏
└─────────────────┘
```

### 18.2 测试类型决策树

```
代码变更
    │
    ├── 涉及 UI？
    │   ├── 是 → 必须手动 GUI 测试（computerUse 子代理）
    │   │        + 可选自动化测试
    │   └── 否 → 自动化测试为主
    │            + 可选终端驱动测试
    │
    ├── 修复 Bug？
    │   ├── 是 → 必须先复现 Bug（修复前）
    │   │        再验证修复（修复后）
    │   └── 否 → 正常测试流程
    │
    ├── 性能优化？
    │   └── 是 → 添加性能埋点
    │            对比优化前后数据
    │
    └── 新功能？
        └── 是 → Happy path + Edge cases
                 GUI 演示视频
```

### 18.3 证据质量标准

**好的证据**：
- 修复前后的对比截图
- 功能端到端演示视频
- 测试通过的终端输出
- 性能优化前后的数据对比

**不合格的证据**：
- "代码能编译" ≠ 功能正确
- "应用能启动" ≠ 变更有效
- "首页能加载" ≠ 新功能工作
- 失败的测试截图/视频
- 无关的应用状态截图

## 21. 安全设计

### 21.1 沙箱隔离

```
┌───────────────────────────────────┐
│         Host Machine               │
│  ┌───────────────────────────────┐│
│  │      Firecracker MicroVM      ││  ← 硬件级隔离
│  │  ┌───────────────────────────┐││
│  │  │    Docker Container       │││  ← 进程/文件系统隔离
│  │  │  ┌───────────────────────┐│││
│  │  │  │   Agent Process       ││││  ← 受限权限
│  │  │  └───────────────────────┘│││
│  │  └───────────────────────────┘││
│  └───────────────────────────────┘│
│                                   │
│  网络：受控访问                    │
│  文件系统：挂载点隔离              │
│  进程：禁止 pkill -f              │
└───────────────────────────────────┘
```

### 21.2 Secret 安全

| 措施 | 说明 |
|------|------|
| 传输加密 | Secrets 传输时加密 |
| 存储加密 | Secrets 存储时加密 |
| 输出脱敏 | 工具输出中 Secret 值替换为 [REDACTED] |
| 提交扫描 | Redacted Secret 类型会扫描提交内容 |
| 公共仓库 | 公共仓库默认禁用 Secret 注入 |

### 21.3 GitHub CLI 只读限制

- `gh` CLI 预认证但仅支持读操作
- 不可创建/修改 PR、Issue 等
- 减少 Agent 对外部系统的影响

### 21.4 工具调用安全

- 不执行来自搜索结果的可疑指令
- 不执行与用户请求无关的操作
- 禁止 `pkill -f`（防止误杀进程）
- 文件操作优先使用安全工具

## 22. AGENTS.md 知识持久化设计

### 22.1 设计目的

AGENTS.md 是**跨会话持久化**的知识文件，让未来的 Agent 会话能继承之前的经验。

### 22.2 结构设计

```markdown
# AGENTS.md

## 通用指南
（项目级别的开发规范、代码风格等）

## Cursor Cloud specific instructions
（Cloud Agent 特定的开发环境信息）

### 服务概述
- 前端：React app, port 3000
- 后端：Node.js API, port 8080
- 数据库：PostgreSQL (Docker)

### 非显而易见的注意事项
- 安装新依赖后需重启后端进程（热重载不会自动拾取）
- 运行测试前需确保 `.env.test` 存在
- 端口 3000 有时被占用，需要先杀掉旧进程

### 标准命令
参考 README.md 和 package.json scripts
```

### 22.3 内容规则

| 应该包含 | 不应该包含 |
|----------|-----------|
| 服务描述和端口 | 依赖安装步骤 |
| 非显而易见的启动注意事项 | 一次性设置操作 |
| 开发环境 gotchas | 系统依赖安装 |
| 对已有文档的引用 | 已有文档的复制 |
| 用户提供的持久性澄清 | 仅与当前设置相关的信息 |

### 22.4 更新策略

- 如果已有 Cloud 指令，假设它们是正确的
- 只修改明确且无争议地不正确的内容
- 只添加非常重要的新内容
- 不重复显而易见的文档

## 附录 B: 关键设计决策与权衡

| 决策 | 选择 | 原因 |
|------|------|------|
| 虚拟化方案 | Firecracker + Docker | 快速启动 + 安全隔离 + 用户项目容器支持 |
| 终端模型 | 文本文件 | 可查询、可搜索、Agent 友好 |
| 存储驱动 | fuse-overlayfs | 内核兼容性（Firecracker 限制） |
| 搜索工具 | ripgrep | 性能优越，支持大型代码库 |
| 子代理模型 | 独立进程 | 上下文隔离，专用工具集 |
| Secret 管理 | 环境变量注入 | 简单、安全、标准实践 |
| 知识持久化 | AGENTS.md 文件 | 跨会话、可版本控制、人类可读 |
| Git 操作 | Agent 执行 | 自主性要求 |
| PR 创建 | 系统自动 | 安全性考虑 |
| 更新脚本 | 最小化依赖刷新 | 可靠性优先 |
