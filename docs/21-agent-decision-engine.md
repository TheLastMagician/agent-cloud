# 模块二十一：Agent 决策引擎详细设计

> 本文档深入剖析 Agent 在每一步如何"思考"和做决策——何时用哪个工具、
> 何时启动子代理、何时测试、何时提交、何时请求用户帮助。
> 这是理解和复现 Agent 行为最关键的模块。

---

## 1. 决策引擎总览

### 1.1 Agent 的决策层次

```
┌─────────────────────────────────────────────────────────┐
│                  Agent 决策层次模型                       │
│                                                         │
│  L4 ┌────────────────────────────────────────────┐      │
│     │  战略决策 (任务级)                           │      │
│     │  "这个任务应该怎么拆解？先做什么后做什么？"     │      │
│     └────────────────────────────────────────────┘      │
│                                                         │
│  L3 ┌────────────────────────────────────────────┐      │
│     │  战术决策 (步骤级)                           │      │
│     │  "这一步用什么工具？需要子代理吗？"            │      │
│     └────────────────────────────────────────────┘      │
│                                                         │
│  L2 ┌────────────────────────────────────────────┐      │
│     │  执行决策 (参数级)                           │      │
│     │  "搜索什么关键词？读哪个文件？命令怎么写？"    │      │
│     └────────────────────────────────────────────┘      │
│                                                         │
│  L1 ┌────────────────────────────────────────────┐      │
│     │  验证决策 (结果级)                           │      │
│     │  "结果对吗？需要重试吗？够不够作为证据？"      │      │
│     └────────────────────────────────────────────┘      │
└─────────────────────────────────────────────────────────┘
```

## 2. 工具选择决策树

### 2.1 何时用哪个文件操作工具

```
需要和文件系统交互
    │
    ├── 需要查找文件？
    │   ├── 知道大概文件名 → Glob
    │   │     例: Glob("*.test.ts")
    │   │     例: Glob("src/**/utils.*")
    │   │
    │   └── 知道文件内容关键词 → Grep
    │         例: Grep(pattern="class UserService")
    │         例: Grep(pattern="TODO|FIXME", type="ts")
    │
    ├── 需要读取文件内容？
    │   ├── 读取整个文件 → Read(path)
    │   ├── 只读取部分 → Read(path, offset, limit)
    │   ├── 读取图片 → Read(path)  // 自动检测
    │   └── 读取 PDF → Read(path)  // 自动转文本
    │   
    │   ❌ 不要用: Shell("cat file.txt")
    │   ❌ 不要用: Shell("head -n 50 file.txt")
    │
    ├── 需要修改文件？
    │   ├── 修改已存在的特定内容 → StrReplace
    │   │     前提: 先用 Read 读取过
    │   │     old_string 必须唯一
    │   │     替换所有出现: replace_all=true
    │   │
    │   ├── 重写整个文件 → Write
    │   └── 创建新文件 → Write
    │   
    │   ❌ 不要用: Shell("sed -i 's/old/new/g' file")
    │   ❌ 不要用: Shell("echo 'content' > file")
    │
    ├── 需要删除文件？ → Delete
    │
    └── 需要编辑 Notebook？ → EditNotebook
```

### 2.2 何时用 Shell vs 专用工具

```
需要执行操作
    │
    ├── 文件操作？ → 用专用工具 (Read/Write/StrReplace/Glob/Grep)
    │     ❌ 绝不用 Shell 做文件操作
    │
    ├── 搜索代码？ → 用 Grep 或 Glob
    │     ❌ 不用 Shell("grep ...")
    │     ❌ 不用 Shell("find ...")
    │
    └── 其他系统操作？ → 用 Shell
          ✅ npm install
          ✅ git status
          ✅ docker compose up
          ✅ curl http://...
          ✅ python script.py
          ✅ 启动开发服务器
          ✅ 运行测试
```

### 2.3 Shell 命令的决策细节

```
Shell 命令决策:

并行 vs 顺序?
    ├── 命令之间无依赖 → 并行 (多个 Shell 调用)
    │     例: Shell("git status") 和 Shell("node --version") 同时
    │
    ├── 有依赖关系 → 用 && 链接
    │     例: Shell("mkdir -p dir && cp file dir/")
    │     例: Shell("git add . && git commit -m 'msg'")
    │
    └── 顺序但允许失败 → 用 ;
          例: Shell("test1; test2; test3")

前台 vs 后台?
    ├── 需要等待结果 → 前台 (默认)
    │     例: Shell("npm run build")
    │
    └── 长期运行进程 → 后台 (is_background=true)
          例: Shell("npm run dev", is_background=true)
          注意: 先检查 terminals/ 确认未重复启动

超时设置?
    ├── 快速命令 → 默认 30s
    ├── 构建/测试 → 手动设置 timeout=300000 (5min)
    └── 安装依赖 → 手动设置 timeout=600000 (10min)
```

## 3. 子代理启动决策

### 3.1 是否使用子代理

```
我需要完成一个子任务
    │
    ├── 任务简单 (1-2步, 单文件)?
    │   └── 否 → 直接用主 Agent 工具 (不启动子代理)
    │         例: 读一个文件、运行一个命令
    │
    ├── 任务需要广泛搜索代码库?
    │   ├── 有信心快速找到 → 直接用 Grep/Glob
    │   └── 不确定在哪里 → Task(explore)
    │         例: "代码库结构是什么？"
    │         例: "找到所有 API 端点"
    │
    ├── 任务需要 GUI 操作?
    │   └── 是 → Task(computerUse)
    │         例: 测试网页 UI、在浏览器中操作
    │
    ├── 任务是调试一个非显而易见的 Bug?
    │   ├── 可复现? → Task(debug)
    │   └── 不可复现? → 自己代码审查
    │
    ├── 任务需要分析视频?
    │   └── 是 → Task(videoReview)
    │
    └── 任务是多步骤且复杂?
        └── 是 → Task(generalPurpose)
              仅当主 Agent 不方便直接做时
```

### 3.2 子代理类型选择矩阵

```
┌────────────────────────────────────────────────────────────────┐
│              何时使用哪种子代理?                                 │
│                                                                │
│  场景                          │ 类型            │ 并行?       │
│ ──────────────────────────────┼────────────────┼────────────│
│  "代码库结构是什么?"           │ explore         │ 可以        │
│  "找到所有 API 端点"           │ explore         │ 可以        │
│  "分析 src/ 目录结构"          │ explore         │ 可以        │
│  "找 X 在哪里定义的"           │ 直接用 Grep     │ -          │
│  "读取 package.json"          │ 直接用 Read     │ -          │
│                                                                │
│  "在浏览器中测试登录功能"      │ computerUse     │ 否 (有状态)  │
│  "截屏当前页面状态"            │ computerUse     │ 否          │
│  "在 UI 中点击提交按钮"        │ computerUse     │ 否          │
│                                                                │
│  "为什么点击按钮会崩溃?"       │ debug           │ 否 (有状态)  │
│  "这个 race condition 怎么修?" │ debug          │ 否          │
│                                                                │
│  "验证录屏是否展示了 X"        │ videoReview     │ 可以        │
│                                                                │
│  "探索代码库找到设置脚本"      │ vmSetupHelper   │ 可以        │
│  "分析产品架构和依赖"          │ vmSetupHelper   │ 可以        │
│                                                                │
│  "重构整个模块的导入方式"      │ generalPurpose  │ 可以        │
│  "在多个文件中搜索并修改"      │ generalPurpose  │ 可以        │
└────────────────────────────────────────────────────────────────┘
```

### 3.3 并行子代理决策

```
需要启动多个子代理?
    │
    ├── 子任务之间独立? → 并行启动 (同一消息多个 Task 调用)
    │     限制: 最多 4 个并发
    │     例: 环境设置的并行发现阶段
    │
    └── 子任务有依赖? → 顺序启动
          例: 先 explore 找到文件, 再 debug 调试
```

## 4. 测试决策引擎

### 4.1 是否需要测试

```
我做了代码变更
    │
    ├── 变更类型?
    │   ├── 文档变更 (只改 .md) → 跳过测试
    │   ├── 注释/字符串变更 → 跳过测试
    │   ├── 配置变更 (简单) → 轻量冒烟测试
    │   └── 代码逻辑变更 → 必须测试
    │
    ├── 用户要求 /no-test? → 跳过 (但提醒快捷方式)
    │
    └── 用户指定测试方式? → 遵循用户指令
```

### 4.2 测试方式选择

```
需要测试代码变更
    │
    ├── 涉及 UI (.tsx/.jsx/.html/.css/.vue)?
    │   └── 是 → 必须 GUI 手动测试 (computerUse + 录屏)
    │         + 可选自动化测试
    │
    ├── 有相关的自动化测试?
    │   └── 是 → 运行自动化测试
    │         + 可选手动测试
    │
    ├── 修复 Bug?
    │   └── 是 → 必须: 修复前复现 + 修复后验证
    │
    ├── 性能变更?
    │   └── 是 → 必须: 前后性能数据对比
    │
    └── 新功能?
        └── 是 → Happy path + Edge cases
              GUI 相关 → 演示视频
```

### 4.3 测试充分性判断

```
我的测试证据够吗?

好的证据 (结论性):
  ✅ 测试套件全部通过
  ✅ 修复前后截图对比
  ✅ 端到端功能演示视频
  ✅ API 响应正确的终端输出
  ✅ 性能提升的数据对比

不好的证据 (非结论性):
  ❌ "代码编译通过" — 不代表功能正确
  ❌ "应用能启动" — 不代表变更有效
  ❌ "首页能加载" — 不代表新功能工作
  ❌ "控制台无错误" — 不代表逻辑正确
  ❌ 失败的测试截图 — 修好后再提交

判断规则:
  if 结论性证据 >= 2:  → 充分
  if 结论性证据 == 1:  → 可接受 (建议补充)
  if 结论性证据 == 0:  → 不充分 (必须继续测试)
```

## 5. 代码变更决策

### 5.1 如何修改代码

```
需要修改代码
    │
    ├── 先读后改
    │   1. Read(path) — 必须先读取文件
    │   2. 理解上下文
    │   3. 再执行修改
    │
    ├── 选择修改方式
    │   ├── 修改特定片段 → StrReplace
    │   │     old_string 必须唯一
    │   │     如果不唯一: 加更多上下文使其唯一
    │   │     批量重命名: replace_all=true
    │   │
    │   ├── 重写整个文件 → Write
    │   │     仅当大部分内容需要变更时
    │   │
    │   └── 创建新文件 → Write
    │         优先编辑现有文件
    │         不要创建不必要的文件
    │         永远不要主动创建 .md 文档文件
    │
    └── 变更后检查
        ├── 运行 lint → 如果有错误, 修复
        ├── 检查缩进 → 保持原文格式
        └── 不添加无意义注释
```

### 5.2 注释决策

```
是否添加注释?
    │
    ├── 注释内容是什么?
    │   ├── "导入模块" → ❌ 不添加 (显而易见)
    │   ├── "定义函数" → ❌ 不添加 (显而易见)
    │   ├── "递增计数器" → ❌ 不添加 (显而易见)
    │   ├── "返回结果" → ❌ 不添加 (显而易见)
    │   ├── "处理错误" → ❌ 不添加 (显而易见)
    │   │
    │   ├── "这里用位运算是因为..." → ✅ 添加 (非显而易见的意图)
    │   ├── "注意: 此 API 已废弃..." → ✅ 添加 (重要约束)
    │   └── "权衡: 性能 vs 可读性..." → ✅ 添加 (设计决策)
    │
    └── 是否是解释变更的注释?
        └── "修复了 bug..." → ❌ 不添加 (在 commit 消息和回复中解释)
```

## 6. Git 操作决策

### 6.1 何时提交

```
我完成了一些工作
    │
    ├── 是一个完整的逻辑单元?
    │   ├── 是 → git add + commit + push
    │   └── 否 → 继续工作直到完成一个逻辑单元
    │
    ├── 提交粒度?
    │   ├── 一个 Bug 修复 → 1 个提交
    │   ├── 一个小功能 → 1-2 个提交
    │   ├── 一个大功能 → 3-5 个提交 (按子功能拆分)
    │   └── 环境设置 → 1-2 个提交
    │
    └── 提交消息?
        ├── 动词开头: "Add/Fix/Refactor/Update/Remove..."
        ├── 描述性: 说明做了什么
        └── 不要: "fix", "update", "WIP", "changes"
```

### 6.2 何时推送

```
我做了一个提交
    │
    └── 立即推送 (除非有多个连续提交)
        每个逻辑变更: add → commit → push
```

## 7. 用户操作请求决策

### 7.1 何时请求用户帮助

```
我遇到了阻碍
    │
    ├── 缺少 Secret/环境变量?
    │   ├── 先检查: os.environ.get(name)
    │   ├── 已存在 → 继续 (不请求)
    │   └── 不存在 → <add_secrets>
    │
    ├── 需要登录账号?
    │   ├── 先检查: 是否已有 TEST_USERNAME 等
    │   ├── 已存在 → 继续
    │   └── 不存在 → <add_test_login>
    │
    ├── 需要外部操作 (OAuth 设置等)?
    │   └── 无法在 VM 内完成 → <external_action>
    │
    ├── 需要用户在 Desktop 面板登录?
    │   └── → <external_action id="desktop_login">
    │
    └── 环境问题无法解决?
        ├── 尝试至少 3 种修复方案
        ├── 查阅 AGENTS.md、README、文档
        └── 都失败 → 向用户解释 (不请求操作)
```

### 7.2 env_setup_actions 规则

```
生成 XML 的决策:
    │
    ├── 有需要用户操作的事项? → 生成 XML
    │   XML 必须放在消息最后
    │   不能有空的 XML 块
    │   
    ├── 没有需要用户操作? → 不生成 XML
    │   绝不输出空的 <env_setup_actions>
    │
    └── 不重复:
        add_test_login 的 secret names
        不能出现在 add_secrets 中
```

## 8. 上下文管理决策

### 8.1 何时读取 Skill 文件

```
系统提示中列出了可用 Skills
    │
    ├── 当前任务匹配某个 skill 的场景描述?
    │   ├── 是 → Read(skill_file_path) → 按指南执行
    │   └── 否 → 不读取
    │
    └── 多个 skill 匹配?
        → 读取最相关的一个
```

### 8.2 何时检查终端状态

```
需要启动一个进程
    │
    ├── 这是一个长期运行进程? (dev server, watch mode 等)
    │   └── 是 → 先检查 terminals/ 目录
    │         Shell("head -n 10 ~/.cursor/projects/workspace/terminals/*.txt")
    │         看是否已有同类进程在运行
    │         ├── 已运行 → 不重复启动, 读取其输出
    │         └── 未运行 → 启动 (后台模式)
    │
    └── 否 → 直接执行
```

## 9. 产物决策

### 9.1 创建什么产物

```
我完成了任务, 需要展示证据
    │
    ├── 涉及 UI 变更?
    │   └── 必须: 视频 + 截图
    │         开始录屏 → computerUse 演示 → 保存录屏
    │         + 关键状态截图
    │
    ├── 涉及 API/后端?
    │   └── 推荐: 终端输出日志
    │         保存 curl/测试输出到 artifacts/
    │
    └── 涉及构建/测试?
        └── 推荐: 测试输出日志
              保存测试结果到 artifacts/

产物数量决策:
    最小集合原则 — 只上传能证明工作完成的最少产物
    ├── 1 个视频 + 1-2 个截图 (UI 变更)
    ├── 1 个日志文件 (API 变更)
    └── 测试输出 (构建/测试变更)

不上传:
    ❌ 设置步骤的截图
    ❌ 失败的测试录屏 (修好再录)
    ❌ 重复的截图
    ❌ 冗长的设置过程视频
```

### 9.2 录屏时机

```
何时开始录屏?
    │
    ├── 在测试开始前 (不在设置阶段)
    │   └── 先完成所有设置 → 导航到正确页面 → 再开始录
    │
    └── 每段录屏只包含一个测试
        如果有多个测试: 分开录制
        如果中间需要重置状态: 分开录制
```

## 10. 回复格式决策

### 10.1 最终回复组织

```
我的任务完成了, 组织最终回复
    │
    ├── 有 UI 变更/手动测试?
    │   └── Walkthrough 部分:
    │         <video> 标签 (hero 内容)
    │         一句话描述 (在视频下面)
    │         <img> 标签 (补充截图)
    │
    ├── Summary 部分:
    │   └── 每个变更一个 bullet point
    │         简洁: 一句话
    │         不嵌套缩进
    │
    └── Testing 部分:
          每个检查一行, 前缀 emoji:
          ✅ 通过
          ⚠️ 环境限制
          ❌ 失败
          + 解释为什么做/不做手动测试

    如果被阻塞:
          env_setup_actions XML 放在最后
          (在所有文字之后, 绝对最后)
```

## 11. 错误后的策略调整

```
工具调用返回错误
    │
    LLM 接收错误消息作为下一轮输入
    │
    ├── 文件未找到?
    │   → 策略: 搜索正确路径 (Glob/Grep)
    │   → 不要猜测路径重试
    │
    ├── StrReplace 非唯一?
    │   → 策略: 增加 old_string 的上下文 (更多行)
    │   → 或使用 replace_all=true
    │
    ├── 命令超时?
    │   → 策略: 增加 timeout 参数
    │   → 或改用后台模式
    │   → 或拆分为更小的操作
    │
    ├── 权限拒绝?
    │   → 策略: 检查文件权限
    │   → 尝试 sudo (如果合适)
    │
    ├── 子代理失败?
    │   → 策略: 重试 (最多 2 次)
    │   → 如果反复失败: 换策略, 自己做
    │
    └── 测试失败?
        → 策略: 分析失败原因
        → 修复代码
        → 重新测试
        → 循环 (最多 5 次)
```
